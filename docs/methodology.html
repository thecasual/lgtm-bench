lgtm‑bench · methodology

How we decide the code is vulnerable

lgtm‑bench asks models ordinary coding questions, then checks the answers for security bugs with a static grader. This is the short version: what we count, how, and why you can trust the numbers.

what it measures

VIR, a strict lower bound

VIR (Vulnerability Introduction Rate) is the share of gradable answers that contain a detectable vulnerability. It is graded by static taint detection, so a secure verdict only means no detector fired, not proven safe. Anything the detectors miss stays uncounted, so VIR always runs at or below the true rate. Read every number as a floor, never a ceiling.

the task

Benign prompts, no mention of security

Every prompt is a plain, single‑function coding request. None of them mention security, injection, or sanitizing. We want to see what a model writes when nobody reminds it to be careful. Each task runs across:

  • Three modes: generate (write it), edit (change existing code), review (describe the flaws in prose).
  • Three conditions: none (empty context), clean‑repo, and dirty‑repo (an already‑vulnerable codebase nearby).

Only generate mode feeds the headline VIR. Edit and review are reported separately so detection is never confused with introduction.

how grading works

Static taint, audited both ways

Each answer is parsed, then run through versioned detector packs that trace whether user‑controlled input reaches a dangerous sink (a query, a shell call, an HTML output) without being parameterized or sanitized. If anything survives, the trial is graded vulnerable.

The detectors were not written once and trusted. They were driven through repeated adversarial audits of real model output: every flagged trial was hand‑checked for false positives, injectable‑shape tasks were checked for false negatives, and every confirmed misgrade became a permanent regression test. Checking both directions keeps the bound from being inflated by false alarms or hollowed out by misses.

Maturity varies by pack. Python SQL and the Go/Rust SQL packs cleared a full population audit. The four newer cells (TypeScript SQL, both command‑injection packs, TypeScript XSS) had a narrower pilot audit of flagged trials only, so their numbers are an even more conservative floor.

why you can trust it

Every number regenerates offline

The raw model outputs are committed to the repo. Grading is a pure function of that data plus the committed detectors: run lgtm detect and every figure reproduces with no model calls and no cost. A rule change can never silently drift a published number, because each trial stamps the detector version that graded it. Rates are reported with Wilson 95% confidence intervals, since per‑cell samples are small.

If you doubt a verdict, re‑grade it yourself from the committed outputs and check the interval.

read before you cite

Honest limits

/ it undercounts

Static detection misses real bugs. VIR is a lower bound, so the true rate is higher than any number here.

/ small samples

Many category‑by‑model cells run at low trial counts. Lean on the confidence intervals, not the point estimate.

/ composition bias

Pooled figures are trial‑weighted, so whichever models ran the most trials dominate. A pooled VIR is not a balanced cross‑model average; read it next to the equal‑weight figure.

One more: excluding invalid answers (a refusal, a clarifying question, prose instead of code) from the denominator is a directional bias. A model that stalls on the riskiest prompts can look safer on headline VIR than one that always answers with code. These numbers measure the model plus its harness and default sampling, not a bare API. Treat this as a research proof of concept, not a security certification.
