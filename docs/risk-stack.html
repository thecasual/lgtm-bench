the phrasing tax

The words we type when moving fast read to the model as permission to skip the careful version:

quick onejust make it work don’t overthink itkeep it short nothing fancyshipping today

The model writes the input check on the calm prompt, and deletes it on the rushed one.

the money

The cheap model is the pricier build

A small open‑weight model runs on a laptop and costs almost nothing per token. That part is true. It is also the least‑capable tier, and it ships a vulnerability about 39% of the time, against 0.3% for a strong one. Call it 100 times more likely to hand you code that needs a security fix.

The open label is a red herring here. The big open models, DeepSeek and Qwen at frontier scale, are as safe as anyone’s. These models are simply small, and risk tracks capability. The ones cheap enough to run on your own hardware are the weak ones.

Every one of those is another pass, often a second model doing the scanning and a third doing the fix. The tokens you saved come back several times over.

Model is the biggest lever, and the one you can’t prompt around. A cheap model plus its rework is usually the pricier build. Pick the model first.

the honest one

Brownfield: the bug it won’t fix

Does pointing a model at a messy, already‑vulnerable repo make it write more new bugs? I tested it. At this sample size, no measurable difference (1 of 32 vs 0 of 32). I won’t claim an effect the data doesn’t show.

The real danger runs the other way. When a model edits code that already has a vulnerability, it leaves the bug in place 88% of the time. Even the safest models.

AI won’t flood your legacy code with new bugs. It won’t drain the old ones either. Brownfield stays your problem.

the 12%, made concrete

Asked only to add a limit parameter to one search function in an already‑vulnerable repo, fable also swapped the string‑built query for a parameterized one. Nobody asked it to. That is the rare good case, not the rule.

what it inherited
"...WHERE name LIKE '%" + name_query + "%'"
what fable returned
"...WHERE name LIKE ? LIMIT ?",
("%" + name_query + "%", limit)

The other 88% keep that first line exactly as it was. The limit got added; the injection stayed.

the appsec question

Do citizen developers mean AppSec has to grow? Probably not.

The reflex: a flood of non‑experts shipping AI code means a flood of bugs, so security teams must add headcount to keep up. On the classic bugs this benchmark measures, the data pushes back. Strong models introduce them so rarely that a governed pipeline plausibly ships cleaner code than the median human hand.

1 : 100→1 : 1000?

The bottleneck was never the count of reviewers. One appsec engineer who moves a team onto a strong model, a calm prompt, and a harness that asks for secure code changes the outcome more than a room full of people reviewing output ever could. The leverage is in advising on the inputs, not staffing up to inspect everything that ships.

The trap runs the other way: scaling by reaching for cheap, lightweight models because they are free to self‑host. That is exactly where the rate explodes. You cannot staff your way out of a bad model, and you should not try to scale by shipping a worse one. Three asterisks keep it honest:

/ cheap models

The near‑zero is the strong tier only. Mid is ~12%, cheap ~39%, and tools often pick the cheap one silently.

/ rushed prompts

Phrasing still ~3× even the good models. Ungoverned tone re‑introduces the risk.

/ legacy debt

AI leaves 88% of existing bugs untouched. No new flood, no cleanup either.

One limit outweighs those three. This benchmark tested a narrow, well‑understood slice: three classic injection classes (SQL injection, command injection, cross‑site scripting), on a limited set of tasks. On that slice, strong models look close to solved, and the traditional injection bug is mostly a thing of the past. It says nothing about the security work that is genuinely hard: authentication, authorization, access control, business logic. Models are not reliable there, and that is exactly where a good security engineer still earns their keep, by threat modeling alongside the developer rather than reviewing their string handling.

Don’t grow AppSec to catch the injection bugs a strong model already avoids. Govern the inputs, then spend your scarce security experts where the model is weak: authn, authz, and threat modeling with the developer.

how this was measured

lgtm‑bench asks models plain, security‑neutral requests and grades the code with static taint detectors. Deterministic, free to re‑run. The metric, VIR, is a strict lower bound: it counts only what it can prove, so true rates run higher. Full data, method, and every flagged trial live in the repo. This is a research proof of concept, not a security certification.
